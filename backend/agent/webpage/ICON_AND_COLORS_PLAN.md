# Plan — Disable agent Icon usage + Dynamic (growable) color palette

Status: **implemented** (Parts 1 & 2). Palette persists in section 1's CBOR blob —
no top-level column, so no `check_tables` needed.

Decisions: Q1 → Option A. Q2 → human icon picker stays; only the agent is restricted.
Q3 → palette persists in section 1's CBOR blob (no migration). Q4 → new colors are
**tailwind-only** (`bg-[#hex]`/`text-[#hex]`/`border-[#hex]`); hex in color attributes is
not used. Add a short colors note to the agent's top instruction.

Two independent changes requested:

1. Stop the page-builder **agent** from using the `Icon` component (it can reference
   Iconify sprite ids it has no way to validate). The agent should only use SVGs it
   generates itself via `generate_svg`.
2. Replace the fixed 10-color palette with a **dynamic, growable list**. The frontend
   sends the agent the current palette; the agent may reuse a color by index OR
   introduce a new color; the frontend detects new colors, appends them to the
   palette, and rewrites references to the palette.

---

## Part 1 — Disable Icon for the agent

### Findings
- `generate_svg` already stores the inner markup in `SectionData.Svgs["genix-svg-N"]`
  and returns just that id; the agent references it as `<Icon svg="genix-svg-N" vb=…/>`.
  The agent **cannot fabricate Iconify ids** — it only ever gets `genix-svg-N` ids back.
- The *only* thing advertising Iconify sprite ids (`icon--mdi-check`) to the agent is
  the `Icon` section in `BUILDER_COMPONENTS.md`, surfaced via the prompt component
  listing and `get_component_docs("Icon")`.
- True inline `<svg>` in section HTML is **not** a clean option: the generic native-tag
  renderer would receive lowercased attributes from htmlparser2 (`viewBox`→`viewbox`),
  which is invalid SVG. The `<Icon>` body-injection path (`{@html body}` inside an
  explicit `<svg viewBox={vb}>`) is what makes generated SVG render correctly.

### Decision (recommended: Option A)
**Option A — keep the robust `generate_svg`→`<Icon svg="genix-svg-N">` rendering, but
remove the Icon component from everything the agent sees.** Net effect: the agent can
only ever emit SVGs it generated; the Iconify-sprite capability disappears from its
knowledge. `<Icon>` remains an internal rendering detail (and the human icon picker in
`TextBlockEditor.svelte` keeps working unchanged).

Changes:
- `BUILDER_COMPONENTS.md`: delete the `## Icon` section (lines ~188–210). The component
  listing in the system prompt and `get_component_docs` derive from this file, so Icon
  drops out of both automatically. `components_test.go` updated (7 components, drop the
  Icon assertions).
- `prompts.go` `systemPromptBase`: rewrite the icon vocabulary line. New wording:
  "To add an icon, call `generate_svg`; it returns a `svgId` — reference it exactly as
  the tool tells you. NEVER write Iconify ids (`icon--…`) and NEVER hand-write `<svg>`."
- `generate_svg` keeps returning `{svgId, viewBox, usage}` unchanged.

> Option B (true inline `<svg>`) is possible but needs SVG tags added to `NATIVE_TAGS`
> **and** case-preserving attribute handling in parse/serialize for `viewBox`,
> `preserveAspectRatio`, gradient attrs, etc. More invasive, more bug surface. Not
> recommended unless you specifically want the SVG markup living directly in the HTML.

## Part 2 — Dynamic, growable color palette

### Findings
- Palette is a hardcoded `defaultPalette` at **every** render site (builder route,
  storefront `webpage/routes/+page.svelte`, base, template-preview). It is **never
  persisted or loaded** with page content.
- Colors resolve as `var(--color-N)` (1-based). `generatePaletteStyles(palette)` emits
  `--color-1..N` on the section root. `color`/`background-color`/`border-color`
  attributes are compiled at parse time: an integer `N` → `var(--color-N)`, any other
  value (`#hex`, `red`, `rgb()`) passes through literally (`style-attributes.ts`).
- Tailwind arbitrary classes (`bg-[#aabbcc]`) survive round-trip as `node.css` tokens
  and are generated by UnoCSS at runtime — they render but bypass the palette.

### Design

**Agent contract (system prompt + `pageContext`):**
- The turn's context gains a `PALETTE:` block listing the current colors with indices,
  e.g. `1=#0f172a 2=#1e293b …`.
- Instruction: "Reuse a palette color by index — `color="3"`, `background-color="3"`,
  `border-color="3"`. For a NEW color not in the palette, put the hex directly in the
  attribute — `color="#aabbcc"`. Prefer reusing existing palette colors." (Attributes,
  not tailwind arbitrary classes, are the canonical channel — they already funnel
  through `compileStyleAttributes`. Arbitrary `*-[#hex]` classes are also absorbed as a
  fallback, see below.)

**Frontend: absorb new colors into the palette (`absorbColors`).**
New util walking the returned AST after the agent responds, before sections are stored:
- In `node.style`: replace each hex color literal with `var(--color-N)`, where N is the
  palette index of that color (existing match by normalized hex, else append a new
  entry and use the new index).
- In `node.css`: rewrite arbitrary color classes `bg-[#hex]` / `text-[#hex]` /
  `border-[#hex]` to `…-[var(--color-N)]` (UnoCSS accepts `var()` in arbitrary values),
  growing the palette the same way.
- Returns the (possibly grown) palette + the rewritten AST.

`editorStore` gains reactive `palette` state (seeded from the default, or loaded). The
route passes `editorStore.palette` instead of the const `defaultPalette`. `ConfigTab`
already iterates `palette.colors`, so the swatch grid grows for free.

**Persistence (required — grown colors must resolve in the published storefront):**
- Backend: add `Palette []string` to a page-level spot. Cleanest fit given the
  section-row model: store it on **section 1** alongside the existing whole-page `Css`
  (mirrors how `PageCss` is handled). Add `Palette []string` to `SectionContent` (and a
  paired top-level column if we want it queryable — likely not needed; keep it in the
  CBOR blob on section 1).
- `savePageContent`: write `editorStore.palette.colors` onto section 0's payload.
- `getPageContent` / `parsePageContentRows`: read it back into `editorStore.palette`,
  falling back to the default when absent.
- Storefront loaders (`getStoreWebpage`, CDN snapshot parse): surface the palette so the
  storefront renders with the same `--color-N` definitions. Each storefront render site
  passes the loaded palette instead of its local `defaultPalette`.

### Type cleanup
`ColorPalette.colors` is typed as a fixed 10-tuple in `renderer-types.ts` but is an
8-tuple in `color-palettes.ts` and used as a plain array everywhere. Change to
`colors: string[]` so it can grow.
